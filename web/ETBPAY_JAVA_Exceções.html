Arquitetura Resiliente (Código de Estudo)

Ao abrir o projeto no seu ambiente local, analise como aplicamos os conceitos de resiliência na prática. O código entregue possui três pilares de defesa implementados:

PILAR 01

A Conexão Segura (ConexaoDB.java)

Observe como a classe de conexão isola o bloco try-catch para o DriverManager. Se o driver do SQLite não for encontrado, a JVM não encerra o aplicativo abruptamente, mas retorna um erro elegante e controlado no console.

PILAR 02

O Escudo da POO (Conta.java)

Analise a classe Conta. Nenhuma variável de dinheiro (saldo) está pública. O encapsulamento força o uso de Getters e Setters, garantindo que regras de negócio futuras (como impedir saque negativo) possam ser injetadas sem quebrar o sistema.

PILAR 03

Persistência Blindada (ClienteDAO.java)

Navegue até o método de cadastro. O uso do PreparedStatement bloqueia ataques de SQL Injection, enquanto o bloco de exceção captura especificamente a SQLException, avisando o usuário caso a tabela de clientes ainda não tenha sido criada no banco.

Esquema de Dados (3FN)

Abaixo está o contrato relacional implementado. O script SQL (`clientes.sql`) encontra-se no pacote do projeto para execução local no VS Code.

CREATE TABLE clientes (
  id_cliente INTEGER PRIMARY KEY,
  cpf VARCHAR(11) UNIQUE,
  nome VARCHAR(100)
);

CREATE TABLE contas (
  id_conta INTEGER PRIMARY KEY,
  numero_conta VARCHAR(20),
  id_cliente INTEGER,
  FOREIGN KEY(id_cliente) REFERENCES clientes(id_cliente)
);

Setup do Ambiente

  • JDK 17 ou superior ativo na estação.
  • Extensão **Extension Pack for Java** ativada no VS Code para automação da pasta `bin`.
  • Driver de comunicação `sqlite-jdbc.jar` embutido na estrutura física do projeto.

Diagnóstico de Falha

Se a string de conexão estiver correta e as tabelas do banco de dados ainda não tiverem sido geradas pelo script SQL, a aplicação Java lançará uma exceção do tipo SQLException informando a ausência da tabela alvo ao tentar disparar o comando de persistência.

Dica do Tech Lead: Lembre-se de verificar se as tabelas do banco foram populadas executando o script `clientes.sql` localmente antes de iniciar a depuração.
