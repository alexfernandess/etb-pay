Tática de Prova (Radar da Banca)

Revisaremos conceitos cruciais de Programação Orientada a Objetos e Java de forma acessível. Entenda o que as bancas cobram e o que o mercado aplica no dia a dia do nosso ETB-Pay.

Polimorfismo

"Muitas formas". O sistema deve ser capaz de chamar o método processar() sem precisar saber se é Pix ou Cartão. A banca pode tentar dizer que isso exige modificar o sistema base todo, o que é falso!

Encapsulamento (private)

O modificador private protege seus dados (como o saldo do cliente). A banca tentará afirmar que itens privados são acessados fora da classe, o que não faz sentido, pois eles ficam "presos" dentro dela.

Deploy Web (WAR vs JAR)

Enquanto o .JAR embala bibliotecas normais, o .WAR (Web Application Archive) é feito para embalar seu projeto para servidores web contendo páginas e lógicas juntas.

Questão 01 - Quadrix (POO / Polimorfismo)

No desenvolvimento do ETB-Pay, utilizamos Polimorfismo para processar diferentes tipos de pagamento (Pix, Cartão). Segundo os princípios de POO, novos tipos de pagamento (como Boleto) não podem ser incorporados ao ETB-Pay sem exigir profundas modificações no código do sistema básico que já processa as transações.

Gabarito: ERRADO

O Polimorfismo serve exatamente para o oposto: permitir incorporar novos objetos (como Boleto) sem precisar alterar o sistema básico. É o princípio do "Aberto para extensão, fechado para modificação".

Missão Prática ETB-Pay

O seu desafio será mentalizar ou escrever num rascunho como você criaria uma interface Pagamento com um método genérico processar(). Em seguida, imagine criar as classes Pix e Cartao que implementam essa interface. O nosso back-end só precisará chamar pagamento.processar(), sem se preocupar em alterar o "núcleo" se amanhã o banco nos pedir para adicionar o pagamento em Criptomoeda!

Questão 02 - CEBRASPE (Java Web / RESTful)

O ETB-Pay vai fornecer uma API para o nosso front-end mobile consultar os saldos. Se a arquitetura escolhida for criar serviços RESTful, os dados dos clientes devem trafegar exclusivamente no formato XML, sendo cada operação (como saldo e extrato) identificada por uma URL específica.

Gabarito: ERRADO

A palavra "exclusivamente" invalida a questão. Os serviços RESTful podem utilizar XML, mas o mercado atual foca pesado na utilização do formato JSON por ser mais leve e fácil de manusear, além de aceitar outros formatos variados (texto simples, HTML, etc).

Missão Prática ETB-Pay

No projeto, converteremos nossos objetos para enviá-los à tela do celular. Seu desafio conceitual é entender por que nós (e as empresas de mercado) fugimos do XML para ir pro JSON. Tente imaginar como estruturar os dados do cliente ETB de forma rápida (ex: {"nome": "Alex", "saldo": 100}). Isso salvará preciosos milissegundos no processamento da API!

Questão 03 - CEBRASPE (Java Web / Deploy)

Ao final do semestre, o back-end do ETB-Pay precisará ser empacotado para distribuição em um servidor web. A extensão de arquivo padronizada que agrupa classes Java, bibliotecas e outros recursos que constituem um aplicativo Web completo no mundo Java é a .WAR.

Gabarito: CERTO

Correto! O WAR (Web Application Archive) é usado para aplicativos web, possuindo uma estrutura específica como a pasta WEB-INF. O JAR é mais comum para aplicações normais ou bibliotecas (como o driver JDBC que usamos).

Missão Prática ETB-Pay

Pesquise rapidamente a diferença entre exportar um JAR e um WAR na sua IDE favorita. Pense que quando as classes Swing sumirem e a aplicação for pro navegador via Servlet, o empacotamento correto (.WAR) é o que vai permitir que você suba o ETB-Pay para uma nuvem web real, e não fique apenas rodando na máquina de vocês!

Questão 04 - Quadrix (POO / Encapsulamento)

No ETB-Pay, o saldo da conta foi declarado com o modificador private para garantir a proteção. A categoria de visibilidade "private" na obtenção do encapsulamento garante que esses membros somente podem ser acessados fora da classe em que foram declarados.

Gabarito: ERRADO

Inversão total de conceito! Membros declarados como private somente podem ser acessados DENTRO da própria classe onde foram declarados. Para o mundo externo interagir, são criados os métodos públicos Getters e Setters.

Missão Prática ETB-Pay

Imagine as consequências de alguém de fora conseguir mudar o saldo = 1000000; só chamando a variável. O desafio é criar métodos públicos depositar(valor) e sacar(valor) dentro da classe Conta. Eles validarão se o valor não é negativo, garantindo que as regras do banco nunca sejam burladas. O encapsulamento não é preciosismo, é segurança!

Questão 05 - Quadrix (POO / Conceito Básico)

Quando estamos modelando o ETB-Pay, utilizamos o termo classe para definir o molde estrutural. Esse termo, na POO, tem a função basilar de definir os atributos e os comportamentos que serão expostos pelo objeto instanciado na memória do sistema.

Gabarito: CERTO

Questão conceitual direta. A classe atua como o projeto/molde, definindo os dados (atributos) e as ações (comportamentos/métodos) que o objeto real possuirá após ser gerado.

Missão Prática ETB-Pay

Sua missão mental é pensar que ao escrever class Cliente, você está apenas criando um PDF descrevendo o que é ser cliente no banco. Ele não existe no mundo real! No entanto, o comando new Cliente() constrói a "casa" e o coloca no sistema pra valer.

Questão 06 - Quadrix (POO / Herança)

No ETB-Pay, para evitar repetição de código, utilizamos o mecanismo de Herança. Por meio desse relacionamento interclasses, é permitido que a subclasse ContaPoupanca herde de forma automática os atributos e os comportamentos de sua superclasse (classe-pai) Conta.

Gabarito: CERTO

Afirmativa perfeitamente alinhada à definição de Herança. Através do termo extends no Java, a classe filha obtém e estende o que foi desenvolvido na superclasse.

Missão Prática ETB-Pay

Crie a classe genérica Conta que já tenha os campos Agência e Saldo. Depois crie ContaPoupanca extends Conta. Não copie e cole atributos da conta de cima; foque apenas naquilo que a poupança faz diferente, como criar um método exclusivo renderJuros(taxa). Viu como a reusabilidade reduz o trabalho de código?

Questão 07 - Quadrix (POO / Vantagens da POO)

A escolha de arquitetar o ETB-Pay utilizando Programação Orientada a Objetos com Java é defendida pelas gerências por ter a característica de diminuir o tempo do ciclo de desenvolvimento a longo prazo, produzindo um software flexível, confiável e com componentes que podem ser reutilizados em diferentes módulos.

Gabarito: CERTO

Embora a POO possa ter uma curva de aprendizado inicial mais íngreme, suas propriedades (Encapsulamento, Herança, Polimorfismo) focam expressamente em modularização, manutenção e reuso, poupando tempo na refatoração da solução de grandes softwares de mercado.

Missão Prática ETB-Pay

Reflita de forma ampla: Quando desenvolvemos a classe Cliente com todo o rigor do encapsulamento de dados, ganhamos um componente que não serve só pro cadastro. Se amanhã criarmos a funcionalidade de "Empréstimo", você vai apenas reutilizar e "chamar" esse mesmo molde já homologado. Menos falhas, menos tempo recriando a roda!
