Tática de Imersão (Radar da Banca)

Revisaremos conceitos cruciais de SGBDs e Saneamento de Dados. Entenda o que as bancas cobram e o que o mercado aplica no dia a dia do nosso ETB-Pay.

DDL vs DML

DDL (Definition) estrutura o "cofre" (CREATE, DROP, ALTER). DML (Manipulation) movimenta o "dinheiro/dados" (INSERT, UPDATE, DELETE). Não confunda DROP com DELETE!

Restrições (Constraints)

O SGBD deve garantir a integridade. A PRIMARY KEY é única e não-nula. A cláusula CHECK defende o banco de lixo (Integridade de Domínio), barrando inserts inválidos.

Limpeza em Voo (ETL)

Ao projetar dados (SELECT), funções escalares como REPLACE() alteram apenas o resultado exibido temporariamente. Para gravar as limpezas de ETL na tabela nova, combinamos com INSERT INTO.

Transações ACID

Atomicidade (Tudo ou nada), Consistência (Respeita as regras), Isolamento (Invisível sem Commit), Durabilidade (Gravou, não perde). O ROLLBACK é seu botão de pânico.

Questão 01 - CEBRASPE (Comandos SQL / DML vs DDL)

Durante a operação de quarentena do ETB-Pay, o engenheiro de dados precisou apagar uma tabela completamente para recriá-la. O comando DROP TABLE pertence à sublinguagem de Manipulação de Dados (DML), pois sua função é apagar os dados armazenados de forma irreversível.

Gabarito: ERRADO

O comando DROP atua na estrutura do banco de dados (apagando a "caixa" ou "cofre"), portanto pertence à DDL (Linguagem de Definição de Dados). A exclusão de linhas específicas (os dados) é feita com DELETE, que aí sim pertence à DML.

Missão Prática ETB-Pay

No projeto real, nós utilizamos `DROP TABLE IF EXISTS tb_clientes;` no script inicial da Aula 2. Isso destrói a tabela antiga e recria um cofre novo e limpo. Se fôssemos usar DML, faríamos apenas `DELETE FROM tb_clientes;`, preservando a estrutura.

Questão 02 - Quadrix (Restrições / Constraints)

Ao projetar o banco de dados do ETB-Pay, o Arquiteto de Software adicionou a restrição de PRIMARY KEY no campo id_cliente. Essa declaração garante, intrinsecamente, que o campo não aceitará valores nulos (NOT NULL) e que cada valor armazenado será exclusivo (UNIQUE).

Gabarito: CERTO

Por definição teórica e prática dos SGBDs relacionais, a Chave Primária (PK) é exatamente a união de duas constraints: UNIQUE (para impedir duplicidade) e NOT NULL (pois todo registro precisa ter uma identificação obrigatória).

Missão Prática ETB-Pay

No projeto ETB-Pay, nós fomos além. Além da `PRIMARY KEY` na coluna ID, nós adicionamos `UNIQUE` na coluna `email`. Qual a diferença? Uma tabela pode ter vários `UNIQUE`, mas apenas UMA `PRIMARY KEY`.

Questão 03 - CEBRASPE (Expressões / GLOB vs LIKE)

Durante a blindagem do cadastro de celulares, utilizou-se a cláusula CHECK associada ao operador GLOB. Diferente do operador LIKE, o GLOB no SQLite é case-sensitive (distingue maiúsculas e minúsculas) e permite o uso avançado de listas e intervalos de caracteres, como [0-9].

Gabarito: CERTO

Exatamente. O LIKE é mais simples, usando '%' e '_'. O GLOB herda o comportamento do Unix wildcarding (RegEx básico), sendo extremamente poderoso para validar classes de caracteres, como números puros `[0-9]`.

Missão Prática ETB-Pay

Lembre-se da nossa aula: O `LIKE` atua como o segurança da balada (avaliação simples), já o `GLOB` é o leitor biométrico. Foi ele que nos permitiu usar `NOT telefone GLOB '*[^0-9]*'` para bloquear o aluno que tentou digitar "(61)" com parênteses.

Questão 04 - Quadrix (DML / Auto Incremento)

Ao se utilizar a propriedade AUTOINCREMENT em uma coluna PRIMARY KEY, torna-se estritamente obrigatório inserir o valor da chave manualmente na instrução INSERT INTO para que o banco consiga prosseguir com a gravação dos dados.

Gabarito: ERRADO

A principal vantagem de um atributo Auto-Incremental (como SERIAL no Postgres, IDENTITY no SQL Server ou AUTOINCREMENT no SQLite) é justamente desobrigar o programador de passá-lo manualmente no INSERT. O SGBD assume o controle gerando números sequenciais automáticos.

Missão Prática ETB-Pay

No projeto, a tabela possui 5 colunas. Porém no nosso `INSERT INTO tb_clientes (cpf, nome, email, telefone)` listamos apenas 4! Ao ignorar a coluna `id_cliente` no comando, deixamos o banco de dados livre para auto-preenchê-la perfeitamente, evitando conflitos (Column Mismatch).

Questão 05 - CEBRASPE (Transações ACID / Atomicidade)

Em um SGBD, o conceito de Atomicidade (propriedade do acrônimo ACID) garante que, se uma transação que envolve a inserção de 10.000 registros falhar na linha 9.999, todas as alterações previamente submetidas nessa mesma transação devem ser automaticamente desfeitas por meio de um mecanismo de rollback.

Gabarito: CERTO

A Atomicidade impõe a regra do "Tudo ou Nada". A transação é indivisível. Se qualquer erro, pane elétrica ou violação de constraint ocorrer antes do COMMIT final, o SGBD é obrigado a reverter ao estado exato pré-transação.

Missão Prática ETB-Pay

É por isso que em nossos scripts robustos abrimos com `BEGIN TRANSACTION;` antes do `INSERT INTO ... SELECT`. Protegemos a base do ETB-Pay para que ela não fique "meio-grávida" de dados se o pipeline do ETL quebrar pela metade.

Questão 06 - Quadrix (Funções Escalares)

A função nativa REPLACE(), quando acionada diretamente numa cláusula SELECT (ex: SELECT REPLACE(cpf, '-', '')), tem a capacidade de alterar permanentemente os dados gravados na tabela física, economizando o uso de comandos de DML (UPDATE).

Gabarito: ERRADO

A cláusula SELECT é apenas uma operação de leitura/projeção. Mesmo usando funções avançadas para modificar visualmente o dado na tela do usuário, nenhuma alteração permanente acontece na base física sem um UPDATE ou INSERT.

Missão Prática ETB-Pay

No lab "O Raio-X", rodamos o SELECT isolado apenas para *homologar* (testar visualmente) como o REPLACE lidava com o lixo. Para tornar a limpeza definitiva na nova estrutura, tivemos que envelopar o comando dentro de um `INSERT INTO tb_clientes ... SELECT ...`!

Questão 07 - CEBRASPE (Engenharia ETL / Mapeamento)

Em uma migração de dados utilizando a instrução INSERT INTO tabela_destino SELECT * FROM tabela_origem, caso a tabela destino possua colunas a mais que a origem, a execução falhará por quebra estrutural ("Column Mismatch"), sendo necessário realizar um mapeamento explícito das colunas alvo e o preenchimento de campos faltantes com literais, como NULL.

Gabarito: CERTO

SGBDs são matemáticos e rigorosos. Um número 'X' de colunas na instrução INSERT exige exatamente o número 'X' de colunas fornecidas pela subquery SELECT, na mesma ordem e tipo de dados compatível.

Missão Prática ETB-Pay

É o clássico erro "3 values for 4 columns". Como `clientes_old` não tinha telefone, tivemos que refatorar a query inserindo propositalmente o comando `NULL AS telefone` no SELECT para a equação fechar com a nova estrutura da tabela blindada.

Questão 08 - Quadrix (DCL e Segurança do SGBD)

Ao se empregar o Princípio do Menor Privilégio na administração do banco de dados, caso um usuário de sistema receba privilégios de gravação (via GRANT INSERT ON tb_clientes), ele herdará, obrigatoriamente e de forma automática, os direitos fundamentais de consulta (SELECT) sobre a referida tabela.

Gabarito: ERRADO

Os privilégios em SQL são atômicos e granulares. O fato de você poder inserir dados em uma tabela não significa que você tenha permissão para ler o que os outros inseriram lá dentro. O DCL exige que os privilégios sejam explicitados individualmente.

Missão Prática ETB-Pay

No laboratório ETB-Pay, isolar acessos garante que, se o microserviço do gateway vazar na internet (hackeado), o invasor apenas conseguirá "inserir lixo" (que ainda seria barrado pelo CHECK). Mas ele jamais conseguirá puxar um SELECT massivo roubando milhões de dados de cartões de clientes.

Questão 09 - CEBRASPE (Tipologia / Valores Nulos)

No contexto do modelo relacional e na linguagem SQL padrão (ANSI), o marcador NULL é processado da mesma forma lógica e computacional que uma string vazia formada por duas aspas simples (''), afinal ambos indicam a ausência de dados válidos.

Gabarito: ERRADO

Erro fatal em Banco de Dados! NULL denota a ausência absoluta de valor (estado desconhecido ou inexistente). Uma string vazia ('') é um dado textual real, que apenas possui comprimento zero, ocupando alocação física diferenciada em disco.

Missão Prática ETB-Pay

Se um cliente não tem telefone cadastrado na base ETB-Pay, forçamos a entrada do valor `NULL` na migração. Gravar um campo como `''` pode acionar lógicas erradas em relatórios futuros e não fará disparar as validações que procuram por "vazio total".

Questão 10 - Quadrix (Transações ACID / Isolamento)

Ao rodar uma transação de Saneamento usando BEGIN TRANSACTION em um ambiente SQLite de rede, um Engenheiro de Dados altera um nome na tabela clientes_old. Devido à propriedade de Isolamento, essas atualizações serão totalmente invisíveis para outras estações de trabalho até que o comando COMMIT seja finalmente processado com sucesso.

Gabarito: CERTO

Afirmativa corretíssima. A propriedade do Isolamento garante que transações concorrentes não afetem umas às outras. A "bolha" temporária que você criou alterando o dado só se tornará o dado oficial da corporação no instante exato do COMMIT.

Missão Prática ETB-Pay

Isso evita o fenômeno da "Leitura Suja" (Dirty Read). Imagina o caos: o Aluno A roda um script corrompido, o Aluno B enxerga isso em tempo real, gera um relatório para o Banco Central... e então o Aluno A digita ROLLBACK. Para evitar esse horror, as transações operam como bolhas lacradas.

Revisão Tática: Banco de Dados & Saneamento (Cebraspe/Quadrix)

Fundamentos de Banco de Dados

  • DDL vs DML: CREATE/DROP/ALTER estruturam o BD (DDL). INSERT/UPDATE/DELETE manipulam os dados gerados (DML).
  • Transações ACID: Atomicidade (Tudo ou nada), Consistência (regras validadas), Isolamento (invisibilidade antes do commit) e Durabilidade (permanência gravada em disco).

Q1 - Comandos SQL / DML vs DDL

  • O que a questão disse: DROP TABLE pertence à DML pois apaga dados. (ERRADO)
  • A Regra (Por Quê): DROP destrói a "estrutura", logo atua na Definição de Dados (DDL). Para apagar os dados internamente preservando a estrutura, usa-se o DELETE (DML).

Q2 - Restrições / Constraints

  • O que a questão disse: PRIMARY KEY garante valores não-nulos e exclusivos. (CERTO)
  • A Regra (Por Quê): A restrição PRIMARY KEY aglutina, de forma implícita e automática, as validações de UNIQUE + NOT NULL na coluna alvo.

Q3 - Expressões / GLOB vs LIKE

  • O que a questão disse: GLOB é case-sensitive e permite validar listas [0-9]. (CERTO)
  • A Regra (Por Quê): Diferentemente da simplicidade do operador LIKE, o GLOB usa a sintaxe de verificação (tipo wildcards Unix) tornando-se poderoso para garantir máscaras rigorosas no banco.

Q4 - DML / Auto Incremento

  • O que a questão disse: Em campos AUTOINCREMENT, torna-se obrigatório inserir o valor da PK manualmente no INSERT. (ERRADO)
  • A Regra (Por Quê): Justamente o oposto. É recomendável ignorar a coluna autoincrementável na instrução de INSERT para que o SGBD cuide da geração paralela, impedindo duplicações.

Q5 - Transações ACID / Atomicidade

  • O que a questão disse: Falhas interrompem todo o lote de registros da transação (Rollback). (CERTO)
  • A Regra (Por Quê): A Atomicidade (O "Tudo ou Nada") garante a reversão ao último estado válido, protegendo sistemas contra quebras parciais ou corrompimentos indesejados.

Q6 - Funções Escalares

  • O que a questão disse: REPLACE no SELECT altera os dados fisicamente. (ERRADO)
  • A Regra (Por Quê): A instrução SELECT é sempre projetiva. Manipulações nela não causam impacto físico na tabela gravada até que estejam atreladas aos comandos nativos de DML, como o UPDATE ou o INSERT INTO.

Q7 - Engenharia ETL / Mapeamento

  • O que a questão disse: INSERT INTO... SELECT falhará com colunas incompatíveis sem mapeamento/NULL explícito. (CERTO)
  • A Regra (Por Quê): Conhecido como Column Mismatch. Para evitar erros na migração, as colunas do SELECT (em tipo e volume numérico) devem coincidir assimetricamente com as do INSERT de destino.

Q8 - DCL e Segurança do SGBD

  • O que a questão disse: Receber o GRANT de INSERT assegura acesso automático a permissões de leitura SELECT. (ERRADO)
  • A Regra (Por Quê): Privilégios no SGBD são atômicos. Sob o princípio do Menor Privilégio, garantir um privilégio de escrita a um usuário nunca implicará herança implícita de leitura no conteúdo inserido.

Q9 - Tipologia / Valores Nulos

  • O que a questão disse: NULL é processado lógica e computacionalmente da mesma forma que strings vazias. (ERRADO)
  • A Regra (Por Quê): A string vazia é um valor textual computado com "0 caracteres". O NULL é estado marcante da "ausência" definitiva ou total desconhecimento na tabela, possuindo ramificações distintas nos filtros matemáticos.

Q10 - Transações ACID / Isolamento

  • O que a questão disse: Transações não efetuadas (sem COMMIT) na rede SQLite são invisíveis para outras estações de trabalho. (CERTO)
  • A Regra (Por Quê): Representa o "I" (Isolamento) do ACID, projetado para evitar a Dirty Read (Leitura Suja) que impactaria processos alheios com informações que sequer foram oficializadas e persistidas na base comum.
